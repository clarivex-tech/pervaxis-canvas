# Pervaxis.Canvas — Technical Specification

**Version:** 1.0.0
**Date:** 2026-04-30
**Status:** Draft

---

## 1. Overview

Pervaxis.Canvas is a polyrepo Angular platform framework providing a standardised, opinionated foundation for building enterprise-grade web and mobile applications simultaneously. It is the horizontal crosscutting platform layer consumed by all Pervaxis-generated customer prints.

Canvas is not an application. It is a collection of versioned npm packages that any generated print depends on. It defines architecture contracts, shared platform services, UI component libraries, and Module Federation integration patterns for both web (Angular 18) and mobile (Ionic + Angular 18) surfaces.

---

## 2. Design Principles

- **Polyrepo by default** — Canvas is a standalone repo. Every customer print is a standalone repo. No monorepo coupling between customers or between platform and domain.
- **Web and mobile always together** — Every generated print contains both an Angular web app and an Ionic mobile app sharing domain libraries.
- **Domain isolation** — Business logic lives in shared libs within the print. UI surfaces (web/mobile) consume libs. Neither surface contains business logic.
- **MFE-first** — Web surfaces are Module Federation hosts or remotes. Domain features are independently deployable MFE remotes.
- **Canvas as dependency, not source** — Prints consume Canvas via npm packages. Canvas internals are never copied into prints. Upgrades happen via version bumps.
- **Forge is the only print generator** — No print is handcrafted. All prints are generated by Pervaxis.Forge using Canvas-aligned templates.

## 7. Module Federation Architecture

Canvas uses `@angular-architects/native-federation` (esbuild-based, no Webpack dependency).

### 7.1 Shell Host Configuration

```typescript
// federation.config.ts (generated in web app)
import { withNativeFederation, shareAll } from '@angular-architects/native-federation/config';

export default withNativeFederation({
  shared: {
    ...shareAll({ singleton: true, strictVersion: true, requiredVersion: 'auto' }),
  },
});
```

### 7.2 MFE Remote Registration

Canvas Shell Core loads the MFE registry at runtime:

```typescript
// Loaded from registry service at bootstrap
interface MfeManifest {
  name: string;
  remoteEntry: string;
  exposedModule: string;
  routePath: string;
  permissions?: string[];
}
```

### 7.3 Registry Service

Each customer shell resolves its MFE list from a centralised registry at boot:

```
GET /api/registry/{customer-id}/remotes
→ [ { name, remoteEntry, routePath, permissions } ]
```

Shell dynamically registers routes based on registry response. MFE remotes are loaded lazily on navigation.

## 8. Authentication Architecture

Canvas uses OIDC (OpenID Connect) with the following flow:

1. Shell initialises OIDC client on bootstrap (`@pervaxis/canvas-shell-auth`)
2. Auth context is stored in `@pervaxis/canvas-platform-auth` — shared across web and MFEs
3. MFEs receive auth context via InjectionToken from `@pervaxis/canvas-mfe-contracts`
4. HTTP interceptor in `@pervaxis/canvas-platform-http` injects JWT on every outbound request
5. Ionic mobile app uses the same platform auth library — Capacitor Browser plugin handles OIDC redirect

---

## 9. State Management Architecture

Canvas uses NgRx Signals (Angular 18 signal-based state, no classic actions/reducers for local state).

- **Local MFE state** — Signal stores scoped to each MFE
- **Cross-MFE shared state** — Shared slice in `@pervaxis/canvas-platform-state`, accessed via InjectionToken
- **Event bus** — Typed event bus in `@pervaxis/canvas-mfe-contracts` for cross-MFE communication without tight coupling

## 11. Versioning Strategy

- Canvas packages follow semantic versioning (`MAJOR.MINOR.PATCH`)
- `forge.json` in every print records the Canvas version used at generation time
- `forge upgrade` command bumps Canvas deps in a print and runs migration scripts if breaking changes exist
- Canvas maintains a `MIGRATION.md` per major version

## 12. Integration with Pervaxis.Genesis (.NET Backend)

Canvas prints and Genesis prints are separate repos but follow complementary patterns:

| Concern | Canvas (Angular) | Genesis (.NET) |
|---|---|---|
| Authentication | OIDC client | JWT validation middleware |
| HTTP | `@pervaxis/canvas-platform-http` | `Pervaxis.Genesis.Base` HTTP client |
| Error handling | `@pervaxis/canvas-platform-error` | `GenesisException` hierarchy |
| Configuration | Angular environment files | `IOptions<T>` pattern |
| DI | Angular providers | `IServiceCollection` extensions |

API contracts between frontend and backend are defined in OpenAPI specs committed to a shared contracts repo (separate from both Canvas and Genesis).

## 13. Security Considerations

- All Canvas packages must pass OWASP dependency audits (`npm audit`) in CI
- No secrets ever committed — all runtime config via environment variables or config service
- Content Security Policy headers enforced at shell level
- MFE remote entry URLs validated against registry allowlist before loading
- JWT claims validated client-side before rendering permission-gated components
- Ionic mobile app uses Capacitor secure storage for token persistence (never AsyncStorage)
